## 4. Hero

Файл: `src/components/Hero.tsx`

### 4.1. Проблема

В Hero сейчас используется scroll-linked motion:

```tsx
const { scrollYProgress } = useScroll();
const bgY = useTransform(scrollYProgress, [0, 0.3], ['0%', '15%']);
const textY = useTransform(scrollYProgress, [0, 0.3], ['0%', '8%']);
```

Это создаёт лишние перерисовки, ощущение дополнительной прокрутки и тормозит мобильную версию.

### 4.2. Требование

Убрать:

- `useScroll`;
- `useTransform`;
- `style={{ y: bgY }}`;
- `style={{ y: textY }}`;
- бесконечную анимацию стрелки на мобильной версии;
- всё, что создаёт дополнительный vertical overflow.

### 4.3. Что оставить

Можно оставить:

- начальное появление текста;
- начальное появление изображения;
- статичную декоративную форму;
- кнопку «Выбрать экскурсию»;
- кнопку «Задать вопрос».

### 4.4. CSS-требования

Hero:

```text
overflow-hidden
min-height не должен создавать второй скролл
на mobile допускается min-h-auto вместо min-h-screen
изображения не должны выходить за ширину viewport
```

## 13. Онлайн-оплата Robokassa

### 13.1. Бизнес-логика

Николай работает как самозанятый. На сайте нужны два сценария:

1. Групповые экскурсии по расписанию - покупка билета онлайн.
2. Индивидуальные экскурсии - заявка, подтверждение, затем предоплата или оплата по ссылке.

### 13.2. Когда включать оплату сразу

Оплату сразу включать для:

- групповых экскурсий с фиксированной датой;
- событий с ограниченным количеством мест;
- лекций;
- форматов, где не нужно предварительное согласование.

### 13.3. Когда не включать оплату сразу

Оплату сразу не включать для:

- индивидуальных маршрутов с нестандартным временем;
- школьных групп;
- больших групп;
- горных маршрутов при нестабильной погоде;
- экскурсий, где нужно предварительно уточнить состав группы.

### 13.4. Номенклатура

До подключения оплаты определить номенклатуру услуг.

Примеры:

```text
Услуга экскурсовода: индивидуальная экскурсия «Знакомство с Алуштой»
Услуга экскурсовода: билет на групповую экскурсию «Алушта в кино»
Услуга экскурсовода: лекция «Истории Алушты»
Предоплата услуги экскурсовода по согласованному маршруту
```

Вид деятельности для НПД: гид-экскурсовод.

### 13.5. API-сценарий

Фронтенд:

```text
POST /api/payment/create
```

Backend:

1. Проверяет событие или заявку.
2. Проверяет наличие мест.
3. Создаёт заказ.
4. Формирует платёжную ссылку Robokassa.
5. Возвращает ссылку пользователю.

Callback:

```text
POST или GET /api/payment/result
```

Backend:

1. Проверяет подпись Robokassa.
2. Обновляет статус заказа.
3. Обновляет сделку в CRM.
4. Уменьшает количество доступных мест.
5. Отправляет пользователю подтверждение, если подключена почта или сообщение.

### 13.6. Статусы оплаты

```text
not_required
pending
paid
failed
refunded
cancelled
```

### 13.7. Важные ограничения

- Не хранить секреты Robokassa во фронтенде.
- Не принимать оплату за чужие экскурсии, если сайт работает в логике самозанятого и собственных услуг.
- Не продавать товары или услуги, которые не соответствуют режиму НПД.
- Следить за годовым лимитом дохода по НПД.
- Для возвратов и переносов иметь отдельную страницу правил.

- `/admin`;

### 20.1. Секреты

- данные админки.

- rate limit;
- honeypot;
- логирование;
- защита от повторной отправки;
- CORS только для нужного домена.

### 20.3. Админка

- авторизация;
- защита маршрутов;
- сессии или JWT;
- закрыть индексацию `/admin`;

- заявки реально попадают в CRM;
